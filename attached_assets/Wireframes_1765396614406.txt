# üó∫Ô∏è Maps/GIS Interface Design Specification

## I. Global Screen Aesthetic and Layout

* **System Name:** MDRRMO Pio Duran Maps
* **Layout:** The interface must dedicate maximum screen space to the map itself, treating it as the core element. The map control/layer menu (`Interactive Map`, `Topographic Map`, etc.) should be presented as a clean, dismissible, or collapsible sidebar.
* **The Map Canvas:** The map area must be edge-to-edge, only constrained by the top header and the left-side layer menu.

## II. Sidebar: Layer and View Control (`map.jpg` Enhancement)

The left sidebar serves as the primary control panel for map layers.

* **Aesthetic:** The sidebar should use a semi-transparent, deep-colored background (Glassmorphism effect) to allow the map to slightly show through, adding a sense of depth and modernity.
* **Layer Organization:** The menu items (`Interactive Map`, `Administrative Map`, etc.) must function as toggle buttons, not just simple links.
    * **Active State:** The currently active map layer (e.g., "Interactive Map") must be highlighted vividly using the system's primary accent color (e.g., Teal) with a contrasting icon and text.
    * **Layer Grouping:** Consider adding subtle visual grouping for related layers (e.g., Hazard Maps could be a collapsible group).

### Layer Descriptions and Functionality:

| Layer Name | Design Focus & Purpose |
| :--- | :--- |
| **1. Interactive Map** | Default view. Clean basemap with basic points of interest. Includes a **quick search bar** within the map frame for addresses/landmarks. |
| **2. Administrative Map** | fetch from google drive folder (1Pz2MM0Ge4RPQ6tdUORibYGoeKepJ9RSt) and the subfolder are collapsible at the sidebar |
| **3. Topographic Map** | **Visualization:** Uses subtle, professional contour lines and elevation shading. |
| **4. Land Use Map** | **Color Palette:** Use standard, easily distinguishable GIS colors (e.g., Green for agriculture, Grey for residential, Red for industrial). |
| **5. Hazards Maps (CRITICAL)** | **Layer Stacking:** This must be a group of sub-layers (e.g., Flood-Prone Areas, Landslide Zones). Allows users to toggle **multiple hazard layers simultaneously** for combined risk assessment. |
| **6. Touristic Map** | Displays local attractions with custom, branded pin icons. |
| **7. Street Map** | Focuses on road networks; useful for evacuation route planning. |

## III. üö® Hazard Visualization and Interactivity

This is the most critical feature and must be designed for instant, high-stakes decision-making.

### A. Visualization Techniques

* **Heatmaps and Polygons:** Flood zones, landslide areas, and other hazards must be visualized using semi-transparent, color-coded polygons or heatmaps (e.g., graduated red/orange for high risk) that clearly overlay the street/satellite basemap.
* **Opacity Control:** A small, floating slider control must be available for users to adjust the opacity of the active hazard layer, allowing them to see the underlying street map clearly.
* **Information Popups (Click/Hover):**
    * Clicking a hazard zone must trigger a minimal, high-contrast modal or "card" that pops up with key data (e.g., "Flood Depth Estimate: 2 meters," "Affected Barangays," "Last Assessment Date").

### B. Dynamic Data Pins and Icons

* **MDRRMO Assets:** Emergency centers, equipment storage, and evacuation centers should use distinct, custom icons that are easily recognizable (e.g., a **First Aid Cross** for medical facilities, a **Shelter Symbol** for evacuation points).
* **Status Indicators (Live Feed Integration):** If the system has live equipment status, the map pins for assets must use small color indicators:
    * **Green:** Fully available (e.g., a Rescue Vehicle).
    * **Yellow:** Low fuel/needs maintenance.
    * **Red:** Deployed/Unavailable.

## IV. Floating Map Tools and Search

To keep the UI clean, essential tools are placed as floating widgets over the map.

* **Floating Menu (Top Right):** A small, circular, icon-based menu that holds secondary tools:
    1.  **Measurement Tool (Ruler icon):** Allows users to measure distances (important for response radius).
    2.  **Print/Export (Printer icon):** To quickly generate a PDF of the current map view with active layers.
    3.  **GPS Locator (Target icon):** Quickly centers the map on the user's current location (`MY_LOCATION`).

* **Search Box (Top Center, within the map):**
    * **Functionality:** A fast-performing search that suggests places, addresses, and **MDRRMO assets/documents** as the user types (e.g., typing "Torn" could suggest the "Tornado Contingency Plan" document).

## V. Footer/Status Bar

A narrow bar along the bottom edge of the screen to show:

* **Coordinates:** Real-time display of the cursor's geographical coordinates.
* **Map Data Source:** "Google," "MDRRMO Data," etc.

This design ensures the Maps interface is not only visually appealing but also a mission-critical tool for data analysis and situational awareness during emergency response.

# üìÑ Document File Management Design Specification

## I. Global Screen Aesthetic and Structure

* **Background:** Maintain the consistent deep Navy/Indigo background (`#1A1E32`).
* **Layout:** A two-pane design is essential:
    1.  **Navigation Sidebar (Left):** For folder structure, categories, and quick access links.
    2.  **Content Area (Right):** For displaying documents, files, and images in a scannable grid/list view.

## II. Sidebar: Navigation and Quick Access

The sidebar must provide a clear hierarchical structure for file access.

* **Folder Tree:** Implement an intuitive, modern folder tree structure (like a stylized file explorer).
    * **Main Categories:** `Home`, `TC Bulletin`, `Resolutions and EO`, `BDRRMP 2025`, `Contingency Plan`.
    * **Active State:** The currently selected folder/category must be clearly highlighted with the system's secondary accent color (e.g., Green or Teal) to show the user's location immediately.
* **Actionable Links:** Include dedicated links for **"Drafts"** and **"Archived"** documents for quick filtering.
* **Visual Indicators:** Folders containing unread or recently modified documents can display a small, colored badge for notification.

## III. Content Area: Search, Sorting, and Display

### A. Advanced Search and Filtering

* **Search Bar:** A prominent, clean search bar (`Search Files...`) at the top of the content area.
    * **Intelligent Search:** The system should support basic full-text search and should be optimized to pull metadata from MDRRMO forms and reports.
* **Filter/Sort Controls:** Positioned adjacent to the search bar.
    * **Sort By:** Use a clean dropdown (e.g., **`Sort by Name`**, **`Date Modified`**, **`File Type`**).
    * **View Toggle:** A toggle to switch between **Grid View** (visual icons/thumbnails for browsing) and **List View** (more detail-oriented, tabular view for scanning file properties).

### B. Document Card Design (Grid View)

* **Aesthetic:** Documents are presented as sleek, custom cards, deliberately avoiding generic operating system file icons.
* **File Type Visualization:**
    * **Word/DOCX:** Use a custom icon with the file type extension displayed clearly (e.g., Orange/Amber color theme).
    * **PDF:** Use a custom icon with a red/crimson theme.
    * **Spreadsheet/XLSX:** Use a custom icon with a green theme.
* **Card Information:** Each card displays:
    * File Type Icon/Preview.
    * Full File Name (truncated with ellipsis if too long).
    * Small metadata line (e.g., `Date Modified: 12/10/2025` or `Author: J. Cruz`).
* **Interaction:** A subtle animation or color change on hover, and a click should open a secure in-app preview or initiate download/edit options.

### C. Documents and Images Separation

* **Section Headers:** Clearly separate the `Documents (X)` and `Images (Y)` sections with bold, modern headings.
* **Image Gallery Cards:** The image cards must use a consistent aspect ratio with a subtle, dark overlay showing the file name. The `360` badge (as seen in `documents.jpg`) should be stylized as a prominent, yet non-intrusive, colored tag (e.g., light blue pill).

## IV. File Action Management

When a user selects one or more documents (via a small checkbox on the card/row), a non-intrusive **Action Bar** must appear (e.g., a slim bar that slides down from the top or floats above the content).

* **Action Bar Contents:**
    * **`Download`**
    * **`Share`** (Internal/Email)
    * **`Move to...`** (Folder selector)
    * **`Delete`** (High-contrast red icon/button)

## V. Custom MDRRMO Form Generation (New Feature)

The system should support standardized template use for efficiency.

* **Access:** A prominent, persistent button in the content area (e.g., **`+ GENERATE NEW MDRRMO REPORT`**) using the primary accent color.
* **Action:** Clicking this button opens a structured input form based on standardized templates (e.g., Incident Report Form, Damage Assessment Form). The completed form is then saved directly into the designated file folder.
